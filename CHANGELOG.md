# سجل التغييرات (Changelog)

كل التغييرات الملموسة في منصة IBP، منظّمة حسب المراحل. الصيغة مستلهمة من [Keep a Changelog](https://keepachangelog.com).

## [المرحلة B1] — التسجيل الذاتي وتزويد المستأجر ✅
- **صفحة `/signup` (الواجهة)**: نموذج تسجيل ثنائي اللغة (اسم الشركة/المدير/البريد/كلمة المرور/الباقة) يربط بـ `POST /signup`، يعرض أخطاء الخادم مباشرةً (البريد مستخدم… إلخ)، ويسجّل الدخول تلقائيًا للوحة المستأجر. روابط من اللاندينق بيج (CTA) ومن صفحة الدخول.
- **مُتحقَّق حيًّا في المتصفح:** تسجيل شركة جديدة ⇒ تزويد المستأجر ⇒ دخول تلقائي ⇒ لوحة المستأجر الجديد تعرض هويّة المالك وبيانات صفرية (مستأجر فارغ) دون أخطاء.

### الواجهة الخلفية
- **`POST /signup`** (عام): يُزوّد مستأجرًا جديدًا **ذرّيًا** (transaction) — Tenant (حالة `TRIAL`، تجربة 14 يومًا) + Subscription على الباقة (افتراضي `basic`) + TenantConfig (تفعيل كل خطوط المنتجات) + فرع رئيسي + **دور «مالك الحساب»** بصلاحيات كاملة على كل الموديولز + المدير + **شجرة حسابات قياسية** (مستوى 1/2 مقفل، حساب ضريبة 0203) + مركز تكلفة — ثم يُسجّل الدخول مباشرةً (JWT).
- **العزل مضمون:** التزويد يجري داخل **سياق طلب فارغ** (`RequestContextService.run({})`) فيتخطّى guard العزل ويمرّر `tenantId` صراحةً — المستأجر الجديد لا يرى بيانات غيره (مُتحقَّق).
- **التحقّق:** البريد فريد عالميًا (تكرار ⇒ `409`، لأن تسجيل الدخول يبحث بالبريد عبر المستأجرين)، سياسة كلمات المرور (`400`)، باقة مجهولة (`422`)، كبح إساءة التسجيل عبر `RateLimitService` (scope `signup`).
- اختبار `signup.e2e` (6 حالات: تزويد + وصول فوري + عزل + التكرار/الضعف/الباقة/الاسم). الإجمالي **e2e 121/121** (16 ملفًا).
- متبقٍّ: صفحة تسجيل في الواجهة (تربط بهذا الـ endpoint). توثيق: [`docs/29`](./docs/29-roadmap-next.md).

## [المرحلة A1] — سائق تخزين سحابي متوافق S3 (بلا تبعية SDK) ✅
- **سائق S3-متوافق** (`s3-driver.ts`) يعمل مع **AWS S3 / Cloudflare R2 / MinIO / Alibaba OSS** عبر واجهة واحدة — **توقيع AWS SigV4 يدويًا** (`node:crypto` HMAC) + `fetch` العام، **بلا إضافة أي تبعية** (التزامًا بثبات الحزمة التقنية).
- **روابط موقّتة مباشِرة للدلو** (`direct:true`): البايتات لا تمرّ بالـ API. رحلة الرفع: `upload-url` ⇒ رفع مباشر ⇒ **`POST /documents/:id/confirm`** (HEAD يثبّت الحجم/البصمة). محليًا يبقى الرفع عبر الـ API بلا تأكيد.
- `StorageService` صار واجهة (facade) تختار السائق من `STORAGE_DRIVER`؛ **السلوك المحلي مطابق تمامًا** (افتراضي). عمليات الخادم `put/get/head` تعمل للسحابي أيضًا (فواتير ZATCA).
- متغيّرات `STORAGE_*` (نفس الاصطلاح القائم) + `STORAGE_FORCE_PATH_STYLE`. توثيق: [`docs/21`](./docs/21-document-service.md) §2 + [`.env.example`](./.env.example).
- اختبار `storage-s3.e2e` (8 حالات): بنية SigV4 + حتمية التوقيع + اختيار السائق + خطأ الإقلاع — **كلّها دون شبكة**. الإجمالي **e2e 115/115** (15 ملفًا).

## [تصحيح محاسبي] — المعالجة الضريبية للعمولة في القيد (وفق المعيار السعودي) ✅
- **ضريبة مخرجات الوسيط على العمولة (15%) تُقيَّد الآن كالتزام** `0203 ضريبة القيمة المضافة المستحقة` في قيد JRV (كانت مفقودة من القيد رغم ظهورها على الفاتورة).
- القيد الجديد متوازن: مدين ذمم العملاء = أمانات (T−C−VATₒ) + عمولة C + ضريبة مستحقة VATₒ. أمانات المؤمِّن صارت صافية بعد العمولة وضريبتها.
- توضيح الفصل: **ضريبة القسط** مخرجات شركة التأمين (تُحوَّل أمانةً، على إشعار العميل) · **ضريبة العمولة** مخرجات الوسيط (تُورَّد لـ ZATCA، على فاتورة المؤمِّن).
- `summary` يكشف `outputVatPayable`. حساب COA جديد `0203`. اختبار `finance.e2e` يؤكّد قيد ضريبة العمولة. **e2e 103/103**. توثيق: [`docs/20`](./docs/20-issuance-and-finance-core.md) §4.

## [المرحلة 4ب+] — جاهزية ZATCA Fatoora المرحلة 2 (Integration) ✅
- **تهيئة معزولة لكل مستأجر** (`TenantZatcaConfig`): رقم ضريبي 15 رقماً، حالة تهيئة، مفاتيح/شهادات مشفّرة at-rest (AES-256-GCM)، عدّاد وتجزئة معزولان.
- **خطّ تهيئة 4 خطوات** (`/zatca/onboard/*`): توليد CSR (ECDSA secp256k1) ⇒ تبادل OTP/CCSID ⇒ فحوص الامتثال (3 مستندات UBL) ⇒ PCSID/ACTIVE — عبر `ZatcaGateway` (Sandbox، نقطة تبديل واحدة للإنتاج).
- **مستند فوترة موحّد** (`BillingDocument`): UUIDv4 + عدّاد/تسلسل/تجزئة SHA-256 معزولة بالمستأجر + QR (TLV وسوم 1–7) + UBL 2.1 (JSON) + كل حقول التعريب السعودي.
- **توجيه B2B/B2C**: مقاصة فورية (Clearance) للمنشآت · إبلاغ خلفي (Reporting، طابور Redis، نافذة 24س) للأفراد.
- **دمج ذرّي** في الاعتماد المالي (ISSUED + JRV + COA + مستندات ZATCA ثم التوجيه). تشفير at-rest عبر `CryptoVaultService` و`ZATCA_ENC_KEY`.
- اختبارات: e2e 103/103 (zatca: 8). توثيق: [`docs/28`](./docs/28-zatca-phase2-fatoora.md). الفرع `phase-4b-zatca-p2`.

- كيانات `Slip` (RFQ)، `Quotation` (هجينة: حقول معيارية + نص حر)، `Endorsement`.
- وحدة `underwriting`: إنشاء Slip (بوّابة الالتزام ⇒ الطلب QUOTING)، إضافة عروض، **جدول مقارنة آلي** (يحدّد الأرخص)، **Firm Order** ⇒ الطلب AWARDED. حماية `module.production`.
- واجهة: منضدة `/tenant/slips/[id]` + زر RFQ في قائمة الطلبات.
- **تهيئة بنية المرحلة 4ب** (stubs): `ChartOfAccount` (17 رقماً، مستويات، On/Off‑Balance، قفل 1/2)، `CostCenter`، `Voucher` (JRV/PYV/RCV/DPV)، `Invoice` ضريبية، `DebitNote`/`CreditNote`.
- اختبارات: e2e 36/36. الفرع `phase-4a-underwriting`.
