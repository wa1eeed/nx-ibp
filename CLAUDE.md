# CLAUDE.md — دستور مشروع منصة IBP

> هذا الملف يُقرأ في كل جلسة. التزم به حرفياً. لا تخالف قاعدة دون إذن صريح من المستخدم.

## 1. ما هو المشروع
منصة **SaaS متعددة المستأجرين** لشركات وساطة التأمين في السعودية، تؤتمت دورة العمل كاملةً (مبيعات ← تسعير ← إنتاج ← خدمة ← مطالبات ← مالية)، وتُباع باشتراكات بباقات وصلاحيات قابلة للتحكم.

المرجع الوظيفي الكامل في `BLUEPRINT.md`. خطة التنفيذ في `ROADMAP.md`. نموذج البيانات في `packages/db/prisma/schema.prisma`. رموز التصميم في `DESIGN.md`. **اقرأها كلها قبل أي مهمة.**

> **التصميم المرجعي:** الصور في `design-references/` هي التصميم البصري المعتمد للمنصة. **يجب فتح هذه الصور ومطابقة التخطيط والألوان والأقسام وترتيب الصفحات والشريط الجانبي عند بناء أي واجهة.** تُوثَّق الـ design tokens المستخرجة في `DESIGN.md` وتُطبَّق عبر `apps/web/src/app/globals.css` و`apps/web/tailwind.config.ts` — لا ألوان أو قيم صلبة خارجها.

## 2. الحزمة التقنية (لا تغيّرها دون إذن)
- الواجهة: **Next.js (App Router) + TypeScript + Tailwind**، دعم RTL وثنائية اللغة (ar/en) من البداية.
- الخدمات: **NestJS** (معماري وحدات — module لكل مجال).
- قاعدة البيانات: **PostgreSQL + Prisma**.
- الكاش/الطوابير: **Redis** (BullMQ للمهام والتذكيرات).
- التخزين: **S3‑متوافق** (Presigned URLs فقط).
- التغليف: **Docker**؛ النشر عبر **Kubernetes + Terraform** (حيادي سحابياً: AWS/GCP/Alibaba).

## 3. قواعد غير قابلة للتفاوض

### عزل المستأجرين (Multi‑tenancy)
- كل جدول يحمل `tenantId` ومُفهرس به.
- **كل** استعلام يُفلتر بـ `tenantId` من سياق الطلب (Prisma middleware يفرضه تلقائياً). ممنوع استعلام بلا نطاق مستأجر.
- العزل يُفرض في طبقة التفويض، لا بخفاء المسارات.

### الأمان والامتثال (PDPL + NCA)
- تشفير at‑rest و in‑transit.
- `auditLog` لكل عملية حساسة (إنشاء/تعديل/حذف، توليد رابط ملف، عملية تحقق، اعتماد).
- بيانات الإنتاج الحقيقية داخل المملكة فقط (يشمل النسخ الاحتياطية والـ logs والمرفقات).
- بيئات التطوير ببيانات وهمية فقط؛ التكامل الحكومي عبر Sandbox.
- لا أسرار في الكود — استخدم متغيرات البيئة (`.env`)، ووفّر `.env.example`.

### الصلاحيات (RBAC + Entitlements)
- بُعدان منفصلان: (أ) **الباقة/الـ entitlements** تحدّد الموديولز والحدود المتاحة للمستأجر؛ (ب) **RBAC** يحدّد ما يفعله الموظف داخل الموديولز المتاحة.
- كل endpoint محمي بفحص الاثنين: هل الموديول مفعّل في باقة المستأجر؟ وهل للمستخدم صلاحية CRUD المطلوبة؟

## 4. معايير الكود
- TypeScript صارم (`strict: true`). لا `any` إلا بمبرر موثّق.
- التحقق من المدخلات على حدود الـ API (Zod أو class‑validator). لا تثق بأي مدخل.
- أسماء واضحة، دوال صغيرة، مسؤولية واحدة لكل وحدة.
- اختبارات لكل منطق حرج (صلاحيات، عزل المستأجرين، حسابات مالية).
- معالجة أخطاء موحّدة + استجابات API متّسقة.
- تعليقات بالعربية أو الإنجليزية مسموحة؛ أسماء الكود بالإنجليزية.

## 5. بنية المجلدات (مبدئية)
```
/apps
  /web        Next.js (الواجهات الأربع: super-admin, tenant-admin, employee, client)
  /api        NestJS (modules: auth, tenants, rbac, billing, clients,
              sales, production, service, claims, finance, documents,
              verification, reports, compliance, hr)
/packages
  /db         Prisma schema + migrations + seed
  /shared     أنواع وثوابت مشتركة (product catalog من الملحق 6.A)
/infra        Docker, k8s, terraform
```

## 6. أسلوب العمل
- اتبع `ROADMAP.md` مرحلةً مرحلة. لا تقفز للمرحلة التالية قبل اكتمال الحالية واجتيازها التحقق.
- بعد كل مهمة: لخّص ما بُني، وكيف يُختبر، وما المتبقّي.
- عند أي غموض أو قرار معماري مؤثّر: اسأل المستخدم، لا تفترض.
- لا تُدخل بيانات حقيقية في أي اختبار. استخدم بيانات seed وهمية.
