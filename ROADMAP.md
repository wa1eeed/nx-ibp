> ينفّذها Claude Code مرحلةً مرحلة. لا تنتقل لمرحلة قبل اكتمال السابقة واجتيازها معيار التحقق.
> راجع `CLAUDE.md` و`BLUEPRINT.md` و`docs/OASIS-IBS-REQUIREMENTS.md` قبل كل مرحلة. كل ميزة تخضع لـ "تعريف تمّ" في `CLAUDE.md`.

## المرحلة 0 — التهيئة (Scaffolding)
- إنشاء monorepo (`apps/web`, `apps/api`, `packages/db`, `packages/shared`, `infra`).
- إعداد Next.js (App Router, TS, Tailwind, RTL + i18n ar/en) و NestJS.
- إدخال `schema.prisma`، أول migration، وملف `seed` ببيانات وهمية.
- Docker compose للتطوير (api + web + postgres + redis).
- `.env.example` لكل المتغيرات. لا أسرار في الكود.
- **التحقق:** المشروع يُشغّل محلياً، الواجهة تفتح، الـ API يردّ على `/health`، الترحيل ينجح.

## المرحلة 1 — الأساس: المصادقة وعزل المستأجرين
- مصادقة (JWT/session) + سياق المستأجر في كل طلب.
- **Prisma middleware** يفرض فلترة `tenantId` تلقائياً على كل استعلام.
- نموذج المستأجر/الاشتراك/الباقة الأساسي.
- **التحقق:** مستخدم من مستأجر لا يرى بيانات مستأجر آخر إطلاقاً (اختبار عزل صريح).

## المرحلة 2 — الصلاحيات (RBAC + Entitlements)
- الأدوار والصلاحيات (module × CRUD) + قوالب الأدوار الجاهزة من `BLUEPRINT.md` §4.
- محرّك الـ entitlements: فحص (هل الموديول مفعّل في الباقة؟) + (حدود/حصص).
- Guard موحّد على كل endpoint يفحص RBAC + entitlement.
- شاشة "إنشاء موظف" بمصفوفة الصلاحيات.
- **التحقق:** موظف بلا صلاحية يُمنع؛ موديول خارج الباقة يظهر مقفلاً ويُمنع وصوله بالـ API.

## المرحلة 3 — العملاء + كتالوج المنتجات + النموذج الديناميكي
- سجل العملاء (أفراد/منشآت) + **«Client ID» تجاري مقروء** + تفرّد `crNumber`/`nationalId` لكل مستأجر.
- إدخال كتالوج المنتجات (الملحق 6.A) + `FormSchema` لكل فرع.
- محرّك النموذج الديناميكي (كتل: تابعين/مركبات/مواقع/شحنات).
- **بوّابة الالتزام:** مراجعة العميل المحوَّل من المبيعات وقبوله/رفضه **قبل** السماح بطلب الأسعار (Compliance Officer).
- **التحقق:** اختيار فرع يولّد النموذج الصحيح؛ يُحفظ الطلب بكتله المتكررة؛ لا طلب أسعار قبل قبول الالتزام.

## المرحلة 4أ — الاكتتاب الفني وعروض الأسعار (RFQ) ✅
- كيانات `Slip` (RFQ) و`Quotation` و`Endorsement`.
- **نموذج إدخال هجين:** حقول معيارية رياضية (Rate/Premium، Deductible، Limit) للمقارنة الآلية + حقل نص حر `generalRemarks` للشروط الاستثنائية.
- **جدول مقارنة آلي** من الحقول المعيارية (يحدّد الأرخص).
- حوكمة: لا Slip قبل `complianceStatus == APPROVED`؛ تنتهي بـ **Firm Order** ⇒ الطلب `AWARDED` (جاهز للإصدار).
- **التحقق:** e2e 36/36 (حوكمة الالتزام + RBAC production + مقارنة + إسناد + عزل).

## المرحلة 4ب — الإصدار والهندسة المالية (Finance Core) ✅
- موافقة فنية ← اعتماد مالي ← `Policy`/`Endorsement` ← ترحيل القيد + إشعار مدين/دائن + فاتورة ضريبية.
- **مولّد تسلسلي بالفرع:** `PREFIX-BRANCH-CLASS-YEAR-SEQ` (1001 لكل مستأجر/فرع/فئة/سنة).
- **شجرة الحسابات (COA) — 17 رقماً:** المستوى 1/2 **مقفل برمجياً** (توحيد تقارير الهيئة)، المستوى 3 **تحليلي ديناميكي** يدعم الرفع الأولي عبر Excel (هجرة الأنظمة القديمة).
- **الفصل الرقابي (On/Off‑Balance):** عزل أقساط العملاء عن عمولات الوسيط (مطلب هيئة التأمين).
- `CostCenter` (3 مستويات تبدأ بالفرع)، السندات `Voucher` (JRV/PYV/RCV/DPV)، `DebitNote`/`CreditNote`، وبنية الفاتورة الضريبية تمهيداً لـ ZATCA.
- فتح حساب تحليلي للعميل في COA عند الاعتماد.
- **التحقق:** صفقة كاملة من الإنشاء حتى القيد المالي، مع تسجيل كل خطوة في `auditLog`.

> بنية 4ب مُهيّأة في المخطط منذ 4أ (ChartOfAccount/CostCenter/Voucher/Invoice/DebitNote/CreditNote)، والمنطق يُبنى في مرحلتها.

## المرحلة 5 — وحدة المستندات ✅
- رفع عبر Presigned URLs، عزل بالمسار `tenant_{id}/`، تسجيل كل رابط في التدقيق.
- ضغط الصور (هدف WebP للمرفقات)؛ استثناء المستندات الرسمية؛ حد الرفع كـ entitlement.
- **التحقق:** الرفع/العرض يعمل بروابط موقّتة فقط؛ لا رابط عام؛ الحد يُطبّق حسب الباقة. (e2e 45/45)

## المرحلة 6 — بقية الموديولز التشغيلية ✅
- خدمة العملاء، المطالبات، التجديدات — على نفس نمط المرحلة 4.
- **التحقق:** كل موديول يتبع نفس الاعتماد والتدقيق والصلاحيات. (e2e 53/53)

## المرحلة 7 — التحقق الحكومي (KYC/KYB) ✅
- طبقة موفّري التحقق (Sandbox أولاً): يقين/واثق/نفاذ/العنوان الوطني + فحص PEP/العقوبات.
- تعبئة ذكية للنموذج، فحص PEP/عقوبات، أرصدة وعدّاد عمليات، نموذجا الفوترة (Pass-through/Reseller).
- **التحقق:** سحب بيانات تجريبي يعبّئ النموذج؛ كل عملية تُخصم وتُسجّل. (e2e 60/60)

## المرحلة 8 — اللوحات والتقارير
> قُسّمت إلى **8أ** (سوبر أدمن) · **8ب** (بوّابة العميل) · **8ج** (التقارير).

### 8أ — لوحة السوبر أدمن ✅
- نطاق منصّة عابر للمستأجرين (`PlatformAdmin` + `scope:platform` + `PlatformGuard`).
- إدارة المستأجرين (تعليق/تفعيل)، الاستخدام الكلّي، والتحكّم في الباقات والـ entitlements والحدود (حدّ الرفع قابل للتهيئة من اللوحة).
- **التحقق:** عزل ثنائي الاتجاه (مستأجر↔منصّة) — e2e 68/68.

### 8ب — بوّابة العميل ✅
- نطاق `client` (ClientUser + scope:client + PortalGuard) بعزل مزدوج (tenantId + clientId).
- صفحات: الرئيسية/الوثائق/الطلبات/المطالبات/كشف الحساب/المستندات (عرض فقط، روابط موقّتة).
- بذرة تشغيلية كاملة + قاعدة اختبار منفصلة (ibp_test).
- **التحقق:** عزل مزدوج مُثبَت — e2e 79/79.
