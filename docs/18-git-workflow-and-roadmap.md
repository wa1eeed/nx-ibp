## 1. سير العمل بالفروع

- **فرع لكل مرحلة** + اختبارات + ملخّص. لا دمج قبل اجتياز التحقّق.
- رسائل الـ commit تنتهي بـ `Co-Authored-By`.
- الفروع الحالية:
  - `phase-0-scaffolding` — يحوي المراحل 0–3 (commit أساس `48601f9`).
  - `phase-4a-underwriting` — المرحلة 4أ + التوثيق الشامل.
  - `phase-4b-finance` — المرحلة 4ب (الإصدار والنواة المالية).
  - `phase-5-documents` · `phase-6-operations` · `phase-7-verification`.
  - `phase-8a-superadmin` — المرحلة 8أ (لوحة السوبر أدمن).
  - `phase-8b-portal` — المرحلة 8ب (بوّابة العميل + قاعدة اختبار منفصلة).
  - `phase-8c-reports` — المرحلة 8ج (التقارير + ربط اللوحة/العمولات ببيانات حيّة).
  - `phase-9-regulatory` — المرحلة 9 (ZATCA + المالية/الالتزام + التكاملات + البنية التحتية).
  - `data-rich-seed` — بيانات شبه واقعية + تحسينات واجهة (تبويبات/بحث/تأكيد/تنقّل/لاندينق بيج).
  - `phase-4b-zatca-p2` — جاهزية ZATCA المرحلة 2 (تهيئة/فوترة/توجيه) + اللاندينق بيج.

## 3. تعريف «تمّ» (Definition of Done)

لكل ميزة (GUIDELINES.md §7):
1. تعمل وظيفياً ومربوطة بقاعدة البيانات.
2. مفلترة بالمستأجر + محمية بـ RBAC والـ entitlement.
3. مُتحقَّق من مدخلاتها + معالجة أخطائها.
4. لها اختبار يغطّي المسار الأساسي والعزل.
5. تُسجّل عملياتها الحسّاسة في `auditLog`.
6. **تحديث التوثيق** في `docs/` (قاعدة دائمة: بعد كل مرحلة تُحدَّث الوثائق المتأثّرة).

## 4. قواعد العمل

- اتبع `ROADMAP.md` مرحلةً مرحلة؛ لا قفز.
- عند أي قرار معماري مؤثّر: اسأل المستخدم.
- بعد كل مهمة: لخّص ما بُني وكيف يُختبر وما المتبقّي.
- لا بيانات حقيقية في أي اختبار؛ بيانات وهمية فقط.
