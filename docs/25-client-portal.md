# 25 — بوّابة العميل (Client Portal)

> المرحلة 8ب: الواجهة الرابعة من المنصّة (بعد السوبر أدمن، tenant-admin، الموظف). يدخلها **عميل شركة الوساطة** (المؤمَّن له) ليتابع وثائقه ومطالباته وطلباته وكشف حسابه ومستنداته — **عرضاً فقط، لا كتابة**. نطاق أمني ثالث (`client`) يخضع لعزل المستأجر ويُضيف فوقه عزلاً بـ `clientId`.

## 1. النطاق الأمني الثالث: `client`

للمنصّة ثلاثة مبادئ أمنية منفصلة لا تلتقي:

| | موظف المستأجر | مدير المنصّة | **عميل البوّابة** |
|---|---|---|---|
| الجدول | `User` | `PlatformAdmin` | **`ClientUser`** |
| نطاق الـ JWT | `tenantId` | `scope:platform` | **`scope:client` + `tenantId` + `clientId`** |
| فلترة Prisma | بـ `tenantId` | غير مفلترة | بـ `tenantId` **+ `clientId` صراحةً** |
| الحماية | `@Authorize` | `PlatformGuard` | **`PortalGuard`** |
| المسارات | `/clients` … | `/platform/*` | **`/portal/*`** |
| الصلاحية | CRUD حسب الدور | إدارة المنصّة | **قراءة بياناته فقط** |

`ClientUser` كيان مستقلّ عن `User` (مبدأ مختلف): يحمل `tenantId` + `clientId` + بريد فريد + `passwordHash`. شركة عميلة قد يكون لها أكثر من مستخدم بوّابة.

## 2. العزل المزدوج (مستأجر + عميل)

بخلاف نطاق المنصّة الذي **يتخطّى** عزل المستأجر، نطاق العميل **يحافظ عليه ويُشدّده**:

1. توكن العميل يحمل `tenantId` ⇒ `TenantContextMiddleware` يضعه في ALS ⇒ Prisma middleware يفلتر كل استعلام بمستأجر العميل (طبقة العزل القائمة، انظر [04](./04-security-and-multitenancy.md)).
2. فوقها، `PortalService` يضيف `where: { clientId }` صراحةً في كل استعلام ⇒ العميل لا يرى إلا بياناته هو، لا بيانات عملاء آخرين في نفس المستأجر.
3. `PortalGuard` يشترط `scope === "client"` ووجود `clientId` ⇒ يمنع موظف المستأجر والسوبر أدمن من المرور.

**العزل مُثبَت بالاختبار من كل الجهات:**
- موظف المستأجر → `/portal/*` ⇒ `403`.
- العميل → مسارات المستأجر (`/clients`) ⇒ `403`.
- عميل مستأجر آخر لا يرى وثائق هذا العميل (يرى وثيقته هو فقط).
- لا يُفتح رابط مستند لعميل لا يملكه ⇒ `404` (فحص ملكية قبل التوقيع).

## 3. الـ endpoints

كلّها بنطاق `client` عدا الدخول. `clientId` يُشتقّ من التوكن (لا يُمرَّر من العميل):

| الطريقة والمسار | الوظيفة |
|---|---|
| `POST /portal/login` | دخول العميل (bcrypt) ⇒ JWT بنطاق `client` |
| `GET /portal/me` | ملف العميل |
| `GET /portal/policies` | وثائق العميل السارية |
| `GET /portal/requests` | طلبات التأمين + طلبات الخدمة |
| `GET /portal/claims` | مطالبات العميل وحالتها |
| `GET /portal/statement` | كشف الحساب: إشعارات المدين + الفواتير + الرصيد المستحق |
| `GET /portal/documents` | مستندات العميل (وثائقه/طلباته/مطالباته) |
| `GET /portal/documents/:id/url` | رابط عرض موقّت (بعد فحص الملكية) |

## 4. الواجهة (Portal)

مسار `/[locale]/portal/*` بـ token مستقلّ (`ibp_portal_token`) ودالة نداء مستقلّة (`cpapi()` في [api.ts](../apps/web/src/lib/api.ts)).

| الصفحة | المحتوى |
|---|---|
| `portal/login` | دخول ⇒ `setPortalToken` ⇒ تحويل إلى لوحة العميل |
| `portal/dashboard` | بطاقات: الوثائق السارية، المطالبات قيد المعالجة، الطلبات المفتوحة، الرصيد المستحق |
| `portal/policies` | جدول وثائق العميل (المنتج، الشركة، الإجمالي، الفترة، الحالة) |
| `portal/requests` | طلبات التأمين + طلبات الخدمة |
| `portal/claims` | مطالبات العميل (المبلغ المطالَب/المُسوّى، الحالة) |
| `portal/statement` | كشف الحساب (إشعارات مدين + فواتير + رصيد) |
| `portal/documents` | مستندات العميل + فتح عبر رابط موقّت |

`PortalShell` ([المكوّن](../apps/web/src/components/portal/PortalShell.tsx)) يغلّف الصفحات بشريط جانبي ووسم «بوّابة العميل»، ويحرس الوصول (لا token ⇒ تحويل إلى `portal/login`). كلّه RTL وثنائي اللغة (namespace `portal`).

## 5. المستندات عبر روابط موقّتة

البوّابة لا تكشف مساراً عاماً لأي ملف. `GET /portal/documents/:id/url`:
1. يجلب المستند ويتحقّق أن `entityId` يخصّ العميل (هو/طلباته/مطالباته/وثائقه) — وإلا `404`.
2. يولّد رابط عرض موقّتاً (5 دقائق، JWT) عبر `StorageService` نفسه المستخدَم في [وحدة المستندات](./21-document-service.md).
3. يسجّل توليد الرابط في `audit_log` (`action: file_url`, `meta.portal: true`).
