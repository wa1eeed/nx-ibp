## A — الإطلاق على Coolify/VPS (P0)
- **A1 — التخزين السحابي الحقيقي** (يفتح الإطلاق): واجهة `StorageProvider` + محوّلات `r2`/`s3` (`@aws-sdk/client-s3`) + رفع مباشر presigned. *(الوضع: التجريد موجود، المحوّلات غير منفّذة.)*
- **A2 — تجهيز النشر**: Docker إنتاجي + env + `migrate deploy` + seed اختياري + `/health` + نطاق/TLS عبر Coolify.
- **A3 — نظافة Git**: إنشاء `main` من `phase-4b-zatca-p2` + رفع للريموت + PRs. *(سريع — يُنفَّذ أولًا.)*

## B — التحويل لمنتج تجاري (P0/P1)
- **B1 — التسجيل الذاتي وتزويد المستأجر**: signup ⇒ إنشاء tenant + اشتراك + أول مستخدم أدمن آليًا (فوق نفس المعمار متعدد المستأجرين).
- **B2 — فوترة الاشتراكات**: ربط بوابة دفع بدورة الباقات (تحصيل/تجديد/تعليق آلي).

## C — الهيكل الإداري والأقسام (P1 — مُوافَق عليه)
- **C1 — كيان `Department` هرمي** (إدارة ← أقسام عبر `parentId`) + `User.departmentId` + صفحة `/tenant/settings/org` تعرض الهيكل كهرم/شجرة.
- **إضافي بحت — صفر تأثير على الصلاحيات** (تبقى على `Role`). **قرار مُوصى:** القسم يحمل **دورًا افتراضيًا اختياريًا** لتسريع تعيين الصلاحيات، مع بقاء الإنفاذ على الدور.

## D — التخزين الاحترافي (P1)
- **D1 — حصص تخزين لكل مستأجر**: هجرة (`plan.storageLimitBytes` + `tenant.storageUsedBytes`) + **حجز ذرّي قبل الرفع** (تجاوز ⇒ 403 + Rollback) + تلميتري في `/admin/plans` (نسبة استهلاك + تنبيه 90% + ترقية يدوية). الحصص على باقاتنا (basic/premium/enterprise).
- **D2 — ضغط Sharp**: تحويل WebP + ضغط 80% + تصغير ≤1200px + `.rotate()` + تراجع سلس.
- **D3 — CDN وسيادة البيانات**: نطاق عام Zero-Egress للأصول العامة + **خيار وجهة تخزين لكل مستأجر** (R2 افتراضيًا · S3-متوافق داخل المملكة عند الطلب لـ PDPL). **قرار مُوصى:** bucket واحد + بادئة `tenant_{id}/` (مع إبقاء bucket-per-tenant خيارًا مؤسسيًا).

## E — تحسينات الأعمال (P2)
- **E1 — ضريبة حسب فرع التأمين** (تأمين الحياة مُعفى 0% بدل 15% الموحّدة).
- [~] **E2 — سلاسل اعتماد قابلة للتهيئة**: ✅ **المحرّك + الـ API** — خطوات اعتماد إضافية (بين الفني والمالي) في `TenantConfig.approvalChains.policy.extraSteps`، لكل خطوة وحدتها/فعلها المطلوب؛ `ConfigService` + `GET/PUT /config/approval-chain` (settings). بعد الموافقة الفنية تُحجَز الخطوات على `Policy.pendingApprovals` (migration `approval_chains`)، و**الاعتماد المالي محجوب** حتى تُعتمد كلها؛ `POST /policies/:id/approve-step` يتحقّق **ديناميكيًا** من صلاحية المستخدم (وحدة/فعل الخطوة) عبر `PermissionService`. افتراضيًا فارغة ⇒ سلوك مطابق للسابق (backward-compatible). اختبار `approval-chain.e2e` (3). [ ] واجهة تهيئة السلسلة + زر «اعتماد الخطوة» على الوثيقة.
- **E3 — تنبيهات** عند انتقالات الحالة (إصدار/اعتماد/تجديد/مطالبة).
- [ ] **E4 — التراجع خطوة للوراء (Undo/Revert)** *(طلب المستخدم — بعد E2)*: صلاحية جديدة `canRevert` (تُمنح لأدوار الإشراف لكل موديول) للتراجع عن **آخر إجراء** على كيان آلة الحالة (Policy/Request/Slip/Claim/ServiceRequest) بإرجاع الحالة خطوة (مع تسجيل `previousStatus`)؛ إجراءات الإنشاء ⇒ إلغاء ناعم؛ **إجراءات مالية/ZATCA ⇒ لا حذف بل إجراء تعويضي** (إشعار دائن/قيد عكسي) حفظًا لثبات السجلّ والتزام الفوترة؛ والتراجع نفسه **إجراء مُدقَّق** جديد. زر «تراجع خطوة» يظهر لمن له الصلاحية. (التفصيل في ذاكرة `ibp-undo-revert-requirement`.)

## G — تحصين أمني والامتثال (P0/P1 — شرط الربط الحكومي)
استيفاء **NCA ECC/CCC/DCC + هيئة التأمين** قبل أي تكامل حكومي/إطلاق حقيقي. التفصيل الكامل والفجوات وخطة المعالجة في [30 — الأمن والامتثال](./30-security-and-compliance.md). أبرزها: rate limiting + قفل الحساب + MFA (P0)؛ refresh tokens + KMS + تشفير PII + مراقبة/WAF + SCA/SAST + نسخ احتياطي/DR + استجابة حوادث (P1)؛ والنقل داخل المملكة + الجانب الحوكمي.

## F — الإنتاج التنظيمي (P2 — يحتاج متطلّبات خارجية)
- **F1 — ZATCA إنتاجي**: استبدال `ZatcaGateway` (Sandbox) بنداءات Fatoora الحقيقية + توليد PDF/A-3 بالختم. *(يلزم اعتماد ZATCA + إنتاج داخل المملكة.)*
- **F2 — BullMQ worker** لطابور الإبلاغ B2C (بدل المُصرِّف الخفيف).
- **F3 — نشر داخل المملكة**: تجسيد وحدات Terraform لمزوّد محدّد + تدقيق أمني + مراقبة/تنبيهات.
