# 33 — دليل الإطلاق (Launch Runbook)

> يحوّل الانتقال من «مكتمل وظيفياً» إلى «مُشغَّل إنتاجياً» إلى **قائمة دقيقة قابلة للتنفيذ**. يُقرأ مع [00 — حالة المشروع](./00-project-status.md) · [32 — البيئات](./32-environments.md) · [`infra/docker/coolify.md`](../infra/docker/coolify.md) · [30 — الأمن والامتثال](./30-security-and-compliance.md).

## 0. مبدأ
النظام مبنيّ بنقاط تبديل (seams): كل تكامل يعمل بـ **Sandbox** ويتحوّل للإنتاج بـ **تغيير متغيّر بيئة + مفتاح** — **دون تغيير كود**. لذا الإطلاق = تهيئة + توصيل + تحقّق، وفق المراحل أدناه.

## 1. قائمة ما قبل الإطلاق (Pre-Launch Checklist)

### أ) البنية والنشر
- [ ] خادم Coolify على **VPS داخل المملكة** (مركز بيانات فئة CST **C** أو أعلى — [30 §5.5](./30-security-and-compliance.md)).
- [ ] قاعدة **PostgreSQL** و**Redis** مُدارتان (نسخ احتياطي مُفعّل) — أو ضمن الحزمة عبر [`docker-compose.coolify.yml`](../infra/docker/docker-compose.coolify.yml).
- [ ] دومينان (الواجهة + الـ API) + شهادات TLS (Coolify تلقائيًا) — TLS 1.3.
- [ ] ضبط أسرار الإنتاج (لا افتراضات): `JWT_SECRET` · `ZATCA_ENC_KEY` · `POSTGRES_PASSWORD` · `CORS_ORIGINS` · `NEXT_PUBLIC_API_URL` · `APP_PUBLIC_URL` · `API_PUBLIC_URL`.
- [ ] الهجرات تُطبَّق تلقائيًا عند الإقلاع (`docker-entrypoint.sh` ⇒ `prisma migrate deploy`).

### ب) توصيل التكاملات (Sandbox ⇒ Production)
| التكامل | متغيّرات الإنتاج | المطلوب منك |
|---|---|---|
| **التخزين السحابي** | `STORAGE_DRIVER=s3\|r2` + `STORAGE_ENDPOINT/BUCKET/ACCESS_KEY/SECRET_KEY/REGION` | دلو **داخل المملكة** |
| **الدفع (Tap)** | `BILLING_GATEWAY=tap` + `TAP_SECRET_KEY=sk_live_…` | حساب Tap إنتاجي |
| **ZATCA Fatoora** | `ZATCA_DEFAULT_ENV=Production` + إتمام خطّ التهيئة الحقيقي (CSR/CSID) | اعتماد الهيئة + شهادات |
| **الإشعارات SMS (Taqnyat)** | `NOTIFY_GATEWAY=live` + `TAQNYAT_API_KEY` + `TAQNYAT_SENDER` | حساب Taqnyat + اسم مُرسِل معتمد |
| **الإشعارات Email (Resend)** | `NOTIFY_GATEWAY=live` + `RESEND_API_KEY` + `NOTIFY_EMAIL_FROM` | حساب Resend + نطاق مُوثَّق |
| **التحقق (يقين/نفاذ)** | مفاتيح المزوّد | حسابات الربط الحكومي |

> **حاليًا:** كل ما سبق يعمل بـ Sandbox؛ توصيل المزوّد الفعلي يتطلّب حساباته/مفاتيحه (خارجة عن الكود). **محوّلات Tap/S3/ZATCA/Taqnyat/Resend جاهزة في الكود** — الإطلاق = إضافة المفتاح وتبديل `*_GATEWAY=live/tap/s3`.

### ج) الأمن والامتثال (NCA ECC/CCC + هيئة التأمين — [30 §5.5](./30-security-and-compliance.md))
- [x] عزل 3 طبقات · RBAC دقيق · **سجل تدقيق ثابت + IP** · **MFA لسوبر أدمن المنصة وللموظفين** (مع إلزام على مستوى الشركة) · تشفير الأسرار.
- [ ] سياسة **الاحتفاظ/الإتلاف الآمن** · **DLP**.
- [ ] **RPO/RTO** (نسخ لحظي + failover ≤ ساعتين، مُختبَر) · **اختبار اختراق** (جهة مرخّصة NCA) · إدارة ثغرات (سدّ 48–72س).
- [ ] فرض احتفاظ السجلّات ≥ سنة + فصل/تشفير خادم السجلّات · KMS/Vault · WAF.
- [ ] بند **حق التفتيش** في العقد الفني + تجهيز تصدير السجلّات (`GET /platform/audit` جاهز).

### د) البيانات
- [ ] **بيانات الإنتاج داخل المملكة** فقط (قاعدة + نسخ + سجلّات + مرفقات).
- [ ] **بذرة إقلاع الإنتاج** (مرجعيات فقط، بلا مستأجرين وهميين): اضبط `PLATFORM_ADMIN_EMAIL`/`PLATFORM_ADMIN_PASSWORD` ثم `pnpm --filter @ibp/db seed:prod` (`SEED_MODE=production`). يبذر الباقات + الكتالوج + المزوّدين + سوبر أدمن فقط. بذرة الديمو مرفوضة تلقائيًا على `NODE_ENV=production` (إلا بـ`SEED_MODE=demo` لبيئة ديمو منفصلة).

## 2. خطوات الإطلاق (Go-Live)
1. **Staging** أولًا: انشر الحزمة على بيئة staging (Sandbox) — [32](./32-environments.md) — واختبر دخانًا/UAT كامل الدورة.
2. **Production**: بعد القبول، انشر بمفاتيح الإنتاج (القسم 1-ب) بموافقة صريحة.
3. تحقّق من `/health` (DB+Redis) و`/health/live`، وتسجيل دخول كل واجهة، ودورة صفقة كاملة تجريبية.
4. فعّل **MFA** لكل حسابات سوبر أدمن المنصة قبل فتح الوصول.

## 3. أوّل حساب حقيقي + الديمو (بعد الجاهزية)
- **إنتاج حقيقي:** بعد `seed:prod` (مرجعيات + سوبر أدمن)، يُنشأ حساب **Gulf Insurance Brokers Co.** عبر `/signup` أو تزويد مباشر ببيانات العميل الحقيقية — انظر [[ibp-launch-and-compliance]] (الذاكرة). **لا بذرة ديمو على الإنتاج الحقيقي.**
- **بيئة ديمو/تجريبية منفصلة** (ليجرّب العميل قبل الالتزام): `seed:demo` يبذر **بيانات ديمو شاملة مترابطة** (عملاء/وثائق/تحقق/تسعير/مالية بكل الحالات + حساب GIB تجريبي) — بيانات وهمية لا PII حقيقي.

## 4. ما بعد الإطلاق
- مراقبة (SIEM/تنبيهات) · نسخ احتياطي مؤتمت + اختبار DR دوري.
- خطة استجابة حوادث + الإبلاغ التنظيمي.
- تدوير الأسرار عبر KMS · مراجعة سجلّات التدقيق دوريًا.

## 5. التراجع (Rollback)
- Coolify يحتفظ بالنشرات السابقة — الرجوع نشرة واحدة. الهجرات **تقدّمية** (لا تراجع تلقائي للبيانات) — خذ نسخة قبل أي إطلاق كبير.
