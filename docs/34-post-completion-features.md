# 34 — طبقة ما بعد الاكتمال (الإشعارات · الضريبة · سلاسل الاعتماد · التراجع · CRM/360°)

> **الدليل السردي الحاكم** للمزايا المُضافة بعد اكتمال النواة (0–9 + ZATCA P2). المرجع البنيوي في [03 — نموذج البيانات](./03-data-model.md) و[06 — مرجع API](./06-api-reference.md) و[07 — الوحدات](./07-backend-modules.md)؛ والحالة/السجل في [00](./00-project-status.md) و[CHANGELOG](../CHANGELOG.md). الأمن في [30](./30-security-and-compliance.md).

## 0. المبدأ
كل ميزة تُبنى بأعلى معيار وساطة تأمين + **حارس امتثالي** (هيئة التأمين/NCA ECC): العزل بالمستأجر، سجل تدقيق ثابت، فصل المهام، صلاحيات دقيقة، ونقاط تبديل (seams) للتكاملات. الاختبارات e2e تغطّي المسار الأساسي + العزل + الحوكمة لكل ميزة.

---

## 1. نظام الإشعارات (لكامل النظام)

### 1.1 البنية
- **بُعد الجمهور (`audience`)**: **client** (يُرسَل لعميل الشركة عبر Email/SMS) و**staff** (يُرسَل لموظفي الشركة عبر البريد — لا هاتف للمستخدمين). إجمالي **18 نوعًا** (7 عملاء + 11 موظفين) في `notifications.constants.ts`.
- **مستويان (`NotificationSetting`)**: **افتراضي المنصة** (`tenantId = NULL`، يديره سوبر أدمن المنصة، يرثه الجميع) و**تخصيص الشركة**. كل نوع قابل للتفعيل/التعطيل وتعديل النص على المستويين. الحلّ: تخصيص الشركة ← افتراضي المنصة ← افتراضي النظام.
- **صفحتا إدارة**: `/admin/notifications` (المنصة) و`/tenant/settings/notifications` (الشركة) — مُجمَّعة بقسمَي «العملاء» و«الموظفين».

### 1.2 التوجيه والإرسال
- **`notify(tenantId, key, to, vars)`** — لعميل (Email/SMS + نسخة in-app إن مُرِّر `clientId`).
- **`notifyStaff(tenantId, key, vars)`** — يوجّه لأصحاب صلاحية **وحدة الحدث** + **مالك الحساب** (أوّل مستخدم) دائمًا، مُزال التكرار (بريد + in-app).
- **`notifyUser(tenantId, userId, key, vars)`** — لمستخدم بعينه (إسناد مهمة/صفقة CRM).
- **بوّابة قابلة للتبديل**: `SandboxNotificationGateway` (تسجيل) · `LiveNotificationGateway` (**Taqnyat SMS + Resend Email**، بلا تبعية) يُفعَّل بـ `NOTIFY_GATEWAY=live` + المفاتيح.

### 1.3 مركز الإشعارات داخل المنصة (in-app)
نموذج `Notification` (per-recipient) يُحفَظ تلقائيًا مع كل حدث مُفعَّل. **جرس `NotificationBell`** مُعمَّم في `Topbar` (الموظف) و`PortalShell` (العميل) — شارة غير المقروء + قائمة + تعليم كمقروء. API: `/notifications/inbox` (موظف) و`/portal/notifications` (عميل).

### 1.4 الأحداث الموصولة (E3)
| الحدث | نقطة الإطلاق | الجمهور |
|---|---|---|
| `policy_issued` · `debit_note` · `request_ack` · `claim_ack` · `renewal_reminder` | إصدار/اعتماد مالي/طلب/مطالبة/تجديد | عميل |
| `staff_request_created` · `staff_quotation_added` · `staff_policy_technical_review` · `staff_policy_finance_review` · `staff_claim_created` · `staff_verification_result` · `staff_wallet_low` · `staff_member_added` · `staff_subscription_status` · `staff_task_assigned` · `staff_deal_assigned` | أحداث التشغيل | موظفون |

كلها **fire-and-forget** — لا تُفشِل العملية الأصل عند تعذّر الإرسال.

## 2. E1 — الضريبة حسب فرع التأمين
تأمين **الحياة (فئة `LIF`)** خدمة مالية **معفاة** من ض.ق.م (المادة 29): **0%** فئة ZATCA **"E"** رمز `VATEX-SA-29-7`؛ بقية الفروع قياسية **15%** فئة **"S"**. يُفرَض عبر `common/tax/vat.ts` (`vatTreatmentForClass`) **عند الإصدار وعند الاعتماد المالي** — حتى لو أُدخل عرضٌ بضريبة خاطئة، ويُوسَم بند ZATCA بالفئة وسبب الإعفاء. **عمولة الوساطة تبقى 15%** (رسم خدمة). المصدر مُطابَق في `@ibp/shared` (`isVatExemptLine`).

## 3. E2 — سلاسل الاعتماد الديناميكية (مع حارس امتثالي)
ديناميكي فيما تسمح به الأنظمة، مع حدٍّ لا يُخالَف. يُهيَّأ من `/tenant/settings/approval-chain` ويُخزَّن في `TenantConfig.approvalChains.policy`:
- **`technicalGate`** — البوّابة الفنية قابلة للتفعيل/التعطيل (عند التعطيل: الإصدار يذهب مباشرة للاعتماد المالي).
- **`extraSteps[]`** — خطوات اعتماد إضافية (لكلٍّ `module/action`) تُحجَز على `Policy.pendingApprovals` عند الموافقة الفنية؛ **الاعتماد المالي محجوب (409)** حتى تُفرَّغ. `POST /policies/:id/approve-step` يفحص الصلاحية **ديناميكيًا** عبر `PermissionService`.
- **`segregationOfDuties`** (افتراضي مُفعَّل — توصية رقابية) — **المعتمِد المالي ≠ مُصدِر الوثيقة** (يُشتقّ المُصدِر من سجل التدقيق الثابت)، قابل للتخفيف للهيكل المرخّص. **الاعتماد المالي/الإصدار خطوة طرفية إلزامية** (يولّد القيد + ZATCA).

## 4. E4 — التراجع خطوة للوراء (Undo/Revert)
- **صلاحية إشرافية `Permission.canRevert`** (فعل RBAC "revert"، رمز مصفوفة "R") — تُمنح للمدير العام/المالك وأي دور مخصّص.
- **`POST /revert/:entityType/:id`** (`RevertService`) — يُرجِع كيان آلة الحالة (policy/claim/service_request/request) خطوة لسابقتها.
- **حواجز امتثالية:** الوثيقة **المُصدَرة (ISSUED) تُرفَض (409)** — تتطلّب إجراءً تعويضيًا (إشعار دائن/قيد عكسي) حفظًا لثبات القيد والتزام ZATCA؛ الحالة الأولى ⇒ 409؛ **التراجع نفسه إجراء مُدقَّق** (لا يمحو التاريخ). زر «تراجع خطوة» على صفحة الوثائق لمن له الصلاحية.

## 5. E5 — إدارة العلاقات (CRM) + رؤية 360°

### 5.1 CRM (`/tenant/crm`)
- **خطّ أنابيب (`Deal`)**: مراحل new→contacted→quoting→proposal→negotiation + كسب/فقد + إسناد + قيمة متوقّعة.
- **مهام/تذكيرات (`CrmTask`)**: أولوية + تاريخ استحقاق + إسناد **يُطلق إشعارًا** للمُسنَد إليه.
- **نشاط/ملاحظات (`CrmActivity`)**: polymorphic؛ نقل المرحلة يُسجَّل تلقائيًا.
- **لوحة «يحتاج متابعة»** (`GET /crm/follow-up`): وثائق قاربت الانتهاء + طلبات مفتوحة + مطالبات نشطة (لمن له claims) + عمولات غير محصّلة (لمن له finance) + مهام متأخّرة — **عابرة للوحدات وتحترم الصلاحيات**.
- **رؤية حسب الدور:** من له **حذف على المبيعات** (مدير) يرى ويعدّل الكل؛ **المندوب** يرى ويعدّل ما أُسنِد إليه أو أنشأه فقط (غيره ⇒ 403). القسم يظهر فقط لمن له وحدة المبيعات.

### 5.2 صفحة العميل 360° (`/tenant/clients/[id]`)
تابات: نظرة عامة (بيانات + تحقّق) · الوثائق المُصدَرة · **التجديدات** (وثائق قاربت الانتهاء ≤90 يومًا بمؤشّر «متبقٍّ/منتهية») · المطالبات · الطلبات · المستندات · كشف الحساب · **الخط الزمني** + إضافة ملاحظة. عبر `GET /clients/:id/overview` المجمّع (معزول بالمستأجر ⇒ عبر مستأجر آخر 404).

### 5.3 صفحة الموظف 360° (`/tenant/settings/staff/[id]`)
بياناته/دوره/قسمه + مؤشرات إنتاجية (عمليات/إصدارات/اعتمادات) + تابات **ما تحت مسؤوليته**: وثائق أصدرها (من التدقيق) · صفقاته (المُسنَدة) · مهامه · النشاط. عبر `GET /staff/:id`.

---

## 6. فلترة العرض حسب الصلاحية
`/auth/me` يُرجِع **خريطة صلاحيات المستخدم** (module ⇒ access/create/edit/delete/revert). **القائمة الجانبية مُفلترة** — كل موظف لا يرى إلا الأقسام المخوّل عليها (دفاع مزدوج مع الـbackend الذي يرفض غير المخوّل بـ403).
