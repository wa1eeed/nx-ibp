# 36 — التحقّق من المعمار والأمن والامتثال (Architecture · Audit · AppSec Verification)

> مرجع تحقّق يربط كل متطلّب من مواصفات **Senior Backend Architect** و**Database Compliance Engineer** و**AppSec/NCA** بتنفيذه الفعلي في الكود (مع الدليل: الملف + اختبار e2e). أُعِدّ ليكون أساسًا لملف العرض الفني لشركات الوساطة. الحالة: **e2e 386/386** (54 ملفًا) · بناءا الإنتاج ✅.
>
> رموز الحالة: ✅ منفّذ ومُختبَر · 🟡 منفّذ بنهج بديل مبرَّر (موثّق) · ⬜ بنية/تشغيلي لاحق (خارج الكود).

## 1. Senior Backend Architect — معمار الـ API

| المتطلّب | الحالة | الدليل |
|---|---|---|
| طبقات خدمة معزولة ومفكوكة (Clean/DDD) | ✅ | معمار NestJS معياري: **Controller → Service → Prisma** لكل وحدة (43 وحدة)؛ المنطق في الخدمات، لا في وحدات التحكّم أو القاعدة. `apps/api/src/modules/*` |
| نقاط نهاية صريحة منظَّمة تطابق مخططات هيئة التأمين/نجم (placeholders) | ✅ | موصّلات التحقّق (`verification.gateway.ts`: يقين/واثق/نفاذ) + تقارير هيئة التأمين (`regulatory.service.ts`) بنمط بوّابة (Sandbox افتراضي · Live BYO) — جاهزة للربط بمخططات المزوّد. |
| منطق الأعمال (إصدار الوثيقة/جلب العروض/تحديث العملاء) عبر وحدات تحكّم متمايزة منفصلة عن القاعدة | ✅ | `production`/`policies` (إصدار) · `slips` (عروض RFQ) · `clients` (تحديث) — كلّها controllers مستقلّة ⇒ services ⇒ Prisma. لا وصول مباشر للقاعدة من الطبقة الشبكية. |
| DTOs صارمة للطلب/الاستجابة بمخططات تحقّق (Zod/Joi/Pydantic equivalent) | ✅ | `class-validator` + `class-transformer` على كل DTO + **ValidationPipe عام** (`whitelist` + `forbidNonWhitelisted` + `transform`) في `main.ts` — يسهّل تخطيط حمولات المؤمِّنين الخارجية. |
| **مستقبِل Webhook موحّد بمصادقة توقيع** لاستقبال تحديثات المؤمِّنين اللاتزامنية بأمان | ✅ | **وحدة جديدة** `modules/webhooks`: `POST /webhooks/carrier/:carrier` عام، مُصادَق بـ **HMAC-SHA256** (`x-carrier-signature`) بمقارنة **ثابتة الزمن** + fail-closed + سرّ من البيئة لكل مؤمِّن. DTO موحّد `CarrierEventDto`. e2e: `security-arch` (توقيع صحيح 200 · خاطئ/غائب 401). |

**ملاحظة معمارية:** الفصل ثلاثي الطبقات (Controller/Service/Data) مع عزل المستأجرين على مستوى Prisma middleware + سياق `AsyncLocalStorage` — يمنع أي تسرّب بيانات عبر الطبقات.

## 2. Database Compliance Engineer — سجل التدقيق المُحكَم (Audit Trail)

جدول `AuditLog` **مستقلّ بنيويًا** عن الجداول التشغيلية ([`schema.prisma`](../packages/db/prisma/schema.prisma)).

| المتطلّب | الحالة | الدليل |
|---|---|---|
| جدول `audit_logs` مركزي مفكوك عن الجداول التشغيلية | ✅ | نموذج `AuditLog` مستقلّ (لا FK تشغيلي عدا `tenantId` للعزل) + فهارس `tenantId`/`createdAt`. |
| هوية الفاعل: `user_id` + `role` + `session_id` | ✅ | `userId` + **`role`** (roleId وقت العملية) + **`sessionId`** (`sid` من JWT، يُولَّد عند تسجيل الدخول). تُلتقط تلقائيًا من سياق الطلب في `AuditService.log`. |
| زمن UTC يولّده الخادم/القاعدة لا العميل | ✅ | `createdAt DateTime @default(now())` — تُولَّد في القاعدة، لا تُرسَل من العميل. |
| شبكة: `ip_address` + `user_agent` + `device_fingerprint` | ✅/🟡 | `ipAddress` (يراعي `X-Forwarded-For` + `trust proxy`) + `userAgent` (بصمة الجهاز على مستوى الخادم). 🟡 بصمة عميل أغنى (canvas/JS) تحسين لاحق. |
| فعل: `action_type` + `target_resource` (جدول + معرّف الصفّ) | ✅ | `action` (create/update/delete/approve/verify/login…) + `entity` + `entityId` = المورد الهدف. |
| لقطتا الحالة: `old_values` + `new_values` (JSON قبل/بعد) | ✅ | حقلا **`oldValues`/`newValues`** (JSONB) + دعم في `AuditService.log`؛ مُطبَّق على تغييرات الحالة (مثل قرار الالتزام). e2e: `security-arch` يثبت التقاط `PENDING⇒APPROVED`. |
| **قيود القراءة فقط** — القراءة لدور المدقّق/سوبر أدمن فقط، لا تعديل/حذف لأي دور، منع تعديل SQL/ORM | ✅ | القراءة محصورة بـ`compliance:read` (شركة) + سوبر أدمن (منصّة). **الثبات (immutability):** يُرفَض `update`/`delete` على `AuditLog` حتى من الكود. e2e: `audit-immutable`. |

**نهج الالتقاط (شفافية):** المنصّة تُدقّق **كل عملية حسّاسة صراحةً** في طبقة الخدمة (أحداث دلالية مثل `POLICY_APPROVED`/`client.approve` بدل فروق صفوف خام مزعِجة) — وهو النهج المُوصى به لسجلّ تدقيق ذي معنى تجاري. يمكن إضافة اعتراض Prisma شامل لكل صفّ كتحسين، مع بقاء الثبات والإثراء الحالي.

## 3. AppSec / Cybersecurity — OWASP Top 10 + ضوابط NCA

| المتطلّب | الحالة | الدليل |
|---|---|---|
| تشفير at-rest (AES-256-GCM) للحقول الحسّاسة | ✅/🟡 | **الآيبان + الجوال** مشفّران (AES-256-GCM، `CryptoVaultService`، e2e `retention-dlp` يُثبت التخزين المشفّر). **بطاقات الدفع لا تُخزَّن إطلاقًا** (توكنة عبر بوّابة Tap) — أقوى من التشفير. 🟡 **الهوية/السجل التجاري** (مفاتيح بحث/تفرّد) تبقى نصًّا لأن تشفير GCM يكسر البحث والتفرّد؛ تُحمى بـ**DLP** (إخفاء لغير الالتزام/المالية) + **تشفير طبقة التخزين (TDE)** في قاعدة الإنتاج المُدارة (blind-index خيار مستقبلي للبحث المشفّر). |
| إدارة مفاتيح آمنة — لا مفاتيح في الكود، بيئة/KMS | ✅ | كل المفاتيح من متغيّرات البيئة (`ZATCA_ENC_KEY`، أسرار Webhook، JWT) — لا سرّ واحد في الكود. جاهز لـKMS في الإنتاج. |
| منع SQL Injection (استعلامات مُعامَلة/ORM آمن) | ✅ | **Prisma ORM** حصريًا (استعلامات مُعامَلة). لا SQL خام في الإنتاج (الاستخدام الوحيد لـ`$queryRawUnsafe` في الاختبارات فقط، مُعامَل بـ`$1`). |
| تعقيم المدخلات + حماية XSS | ✅ | **ValidationPipe** (whitelist) على كل الحدود + الواجهة **React/Next** (تهريب افتراضي) + **CSP** (`frame-ancestors 'none'`، script/style مضبوطة). الـAPI يُعيد JSON فقط (لا HTML). |
| CORS صارم + Rate Limiting على كل نقاط النهاية (ضد DDoS) | ✅ | CORS من البيئة (لا قيم صلبة) + **`ThrottleGuard`** عام لكل IP (Redis، fail-open) **فوق** قفل القوّة الغاشمة للدخول + **حدّ حجم الجسم 2mb**. |
| تجزئة كلمات مرور قوية (Argon2id أو bcrypt بعامل عالٍ) | ✅ | **bcrypt بعامل كلفة 12** (توصية OWASP للكلفة العالية) في كل مواضع التجزئة الثلاثة. |
| ترويسات أمان + إخفاء البصمة | ✅ | HSTS (سنتان+preload) · `X-Frame-Options: DENY` · `nosniff` · `Referrer-Policy` · `Permissions-Policy` · إزالة `X-Powered-By`. |
| مراجعة أمنية (`/security-review`) بعد التوليد | ✅ | **CI أمني آلي** (`security.yml`: Semgrep SAST + gitleaks + pnpm audit) + **Dependabot** (SCA مستمر) + **مراجعة يدوية** لهذه الجلسة (النتائج في §4). `/security-review` التفاعلي متاح لتشغيله في الطرفية. |

## 4. ملخّص المراجعة الأمنية (Security Review) — تغييرات هذه الجلسة

مراجعة يدوية لكل الكود المُولَّد (بديل `/security-review` التفاعلي، مع تعذّر تشغيل أمر CLI تفاعلي كأداة):

- **Webhook:** مقارنة توقيع **ثابتة الزمن** (`timingSafeEqual` + فحص الطول) · **fail-closed** (بلا سرّ/توقيع ⇒ رفض) · السرّ لا يُسجَّل · `eventId` المُعاد مُتحقَّق (JSON، بلا XSS) · لا تلوّث prototype في التسلسل الحتميّ · محميّ بتحديد المعدّل وحدّ الجسم. **لا ثغرات.**
- **التشفير:** AES-256-GCM بمصادقة · المفتاح من البيئة (لا تسريب/تسجيل) · `tryDecrypt` آمن الفشل (يُعيد الأصل بلا انهيار). **لا تسريب.**
- **التدقيق:** `sessionId` عشوائي (12 بايت) غير قابل للتخمين · `oldValues/newValues` الحالية بلا PII (حالة الالتزام) · السجلّ ثابت ومحصور القراءة. **آمن.**
- **الأسرار:** لا أسرار في الكود؛ كلها بيئة. gitleaks في CI يحرس ضدّ التسريب. **نظيف.**

**الخلاصة:** لا ثغرات حرجة، ولا أسرار مسرّبة، ولا كشف بيانات غير آمن في التغييرات.

## 5. المتبقّي (بصدق) — بنية/تشغيلي خارج الكود

- ⬜ **KMS/Vault** لإدارة المفاتيح في الإنتاج (بدل env) · **WAF** · **SIEM/مراقبة وتنبيهات** · **نسخ احتياطي + خطة تعافٍ (DR)** واختبارها · **اختبار اختراق مستقل** · **TDE** لطبقة التخزين للمعرّفات القابلة للبحث · **DAST** في CI. تفصيلها وأولوياتها في [30](./30-security-and-compliance.md).
- 🟡 **blind-index** للبحث المشفّر على الهوية/السجل (بديل لـTDE) · **بصمة جهاز عميل أغنى** · **اعتراض Prisma شامل** للتدقيق التلقائي لكل صفّ.

> هذه بنود بيئة إنتاج داخل المملكة (لا تُنجَز بالكود وحده) وتُفعَّل عند تجهيز نسخة Production المفصولة.
